Socket UML Class Diagram for the C++ Rewrite
===========================================

Purpose
-------
This is a structural UML class diagram for the socket layer rewrite. It focuses on
class responsibilities and relationships such as inheritance, composition,
aggregation, association, and dependency.

Main packages
-------------
[protocol]
[transport]
[dispatch]
[service]

Relationships
-------------
- transport::HttpServer *-- transport::ThreadPoolWorker : creates
- service::PiBService *-- transport::HttpServer : composition
- service::PiBService *-- transport::SocketConnection : composition
- service::PiBService *-- dispatch::DeviceDispatcher : composition
- service::PiBService o-- transport::ThreadPoolWorker : manages
- service::PiAService *-- transport::SocketConnection : composition
- service::PiAService *-- dispatch::DeviceDispatcher : composition
- dispatch::DeviceDispatcher o-- dispatch::IDeviceHandler : registers
- dispatch::WmosHandler ..|> dispatch::IDeviceHandler : implements
- dispatch::UnknownHandler ..|> dispatch::IDeviceHandler : implements
- transport::HttpServer --> protocol::JsonMessage : uses
- transport::SocketConnection --> protocol::JsonMessage : exchanges
- service::PiAService --> protocol::JsonMessage : processes
- service::PiBService --> protocol::JsonMessage : parses and forwards

Design notes
------------
- JsonMessage is the shared value object for all socket communication.
- SocketConnection isolates the TCP details from the service classes.
- DeviceDispatcher owns the runtime mapping from device name to handler.
- PiBService is responsible for HTTP ingress and forwarding to Pi-A.
- PiAService is responsible for peer-side reception and acknowledgement.
- The interface-based handler model replaces the C function table with polymorphism.

Legend
------
- `*--` composition
- `o--` aggregation
- `-->` dependency / usage
- `..|>` interface realization
